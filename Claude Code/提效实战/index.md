# 提效实战

这是本专栏的**核心章节**。Claude Code 的真正威力不在于「能用」，而在于掌握正确的方法后能带来数量级的效率提升。本章汇总了官方最佳实践和社区高手的经验。

## 官方五大最佳实践

Anthropic 在官方文档中明确强调了五个设计原则，理解它们是高效使用的前提。

### 1. 上下文窗口管理（最关键）

> 上下文窗口是你最宝贵的资源，也是最容易被忽视的瓶颈。

**问题**：随着上下文填满，Claude 的表现会下降——可能遗忘早期指令，生成质量降低。一个调试会话就可能消耗数万 token。

**实战策略**：

| 策略 | 做法 |
|------|------|
| **及时清理** | 不相关任务之间用 `/clear` 清空上下文 |
| **精准引用** | 用 `@filename` 直接引用文件，而非让 Claude 自行搜索 |
| **压缩上下文** | 用 `/compact` 命令压缩冗余内容 |
| **知道何时放弃** | 连续纠正两次还不对，开新会话重来 |
| **粘贴原始日志** | 直接粘贴错误日志原文，而非口头描述 |
| **控制模型窗口** | 用 `/model opus[1m]` 扩展上下文到 100 万 token |

::: tip 经验法则
当你发现 Claude 开始「健忘」或重复犯错时，80% 的情况是上下文污染而非模型能力不足。`/clear` 或新会话是最快的解法。
:::

### 2. 提供验证方法（最高杠杆）

> 给 Claude 一种方式来自我检查，是投入产出比最高的实践。

Claude Code 在有明确成功标准时表现显著提升。不要只描述需求，还要告诉它**怎么验证结果是对的**。

**做法**：
- **写好测试**：让 Claude 先写测试再写实现，或指定现有测试命令
- **给出预期输出**：「执行后应该看到 X」「页面上应显示 Y」
- **截图验证**：结合 Playwright MCP 或浏览器工具做视觉验证
- **类型检查**：「修改完后 `tsc --noEmit` 应该零错误」

```
❌ "帮我写一个分页组件"
✅ "帮我写一个分页组件，要求：
   - 通过 src/components/__tests__/Pagination.test.ts 中的所有测试
   - pnpm lint 零警告
   - 支持 totalPages、currentPage、onPageChange 三个 props"
```

### 3. 探索→规划→编码（Explore → Plan → Code）

> 把研究、规划和实现分开，避免解决错误的问题。

**三阶段工作流**：

```
阶段一：探索（Explore）
  "阅读 src/auth/ 目录，理解当前的认证流程"
  "找出所有调用 userService.getProfile() 的地方"

阶段二：规划（Plan）
  按 Shift+Tab 进入 Plan Mode
  "根据刚才的分析，制定重构方案，列出需要修改的文件和步骤"

阶段三：编码（Code）
  按 Shift+Tab 退出 Plan Mode
  "按照上面的方案执行修改"
```

::: warning 常见错误
直接说「帮我重构认证模块」会让 Claude 在探索和编码之间反复切换，浪费大量上下文。**先理解、再规划、最后动手**能显著提高一次性成功率。
:::

### 4. 环境定制

利用 CLAUDE.md、Hooks、Skills、权限白名单等配置能力来减少人工干预。

**权限白名单**：将可信命令加入白名单，消除频繁确认弹窗

```bash
# 通过 /permissions 命令配置
/permissions

# 或在 .claude/settings.json 中设置
{
  "permissions": {
    "allow": [
      "Read",
      "Bash(npm test *)",
      "Bash(git status)",
      "Bash(git diff *)"
    ]
  }
}
```

### 5. 避免已知失败模式

| 失败模式 | 正确做法 |
|---------|---------|
| 一次给太多无关任务 | 一次一个任务，完成后再给下一个 |
| 在上下文污染后继续纠正 | `/clear` 或开新会话 |
| 口头描述错误而非粘贴日志 | 直接粘贴完整错误信息 |
| 让 Claude 猜你的意图 | 明确写出验收标准 |
| 忽视 CLAUDE.md 的维护 | 每次发现 Claude 犯错就更新 CLAUDE.md |
| 使用过于模糊的指令 | 具体化：包含文件路径、函数名、预期行为 |

## 并行工作流（效率提升倍数最大）

**并行是官方推荐的第一大效率提升手段**。核心思路：用 Git Worktree 创建多个独立工作目录，每个运行一个独立 Claude Code 会话。

### Git Worktree 并行开发

```bash
# 创建独立工作树
claude --worktree feature-auth
claude --worktree bugfix-login
claude --worktree refactor-api

# 或手动创建
git worktree add ../project-feature feature-branch
cd ../project-feature
claude
```

**推荐做法**：
- 同时开 3-5 个 Worktree，分别处理不同功能/修复
- 配合 shell 快捷键快速切换（如 `alias za='cd ../worktree-a && claude'`）
- 每个 Worktree 有独立的分支、文件和配置，互不干扰

### 子代理（Subagents）并行

Claude Code 内置多代理编排能力。主代理可以委派任务给子代理，每个子代理有独立的上下文窗口。

```
主代理：「我需要同时完成这三件事」
  ├── 子代理 A：重构 auth 模块
  ├── 子代理 B：编写测试用例
  └── 子代理 C：更新 API 文档
```

子代理的核心优势：
- **速度**：独立任务并行执行
- **隔离**：一个子代理失败不影响其他
- **上下文干净**：每个子代理有自己的上下文窗口，不会污染主线程

## 快捷操作速查

| 操作 | 快捷方式 | 说明 |
|------|---------|------|
| **Bash 直通** | `!` 前缀 | `!git status` — bash 输出直接进入上下文 |
| **后台运行** | `Ctrl+B` | 长任务在后台执行，不阻塞当前会话 |
| **编辑计划** | `Ctrl+G` | 编辑 Claude 的执行计划后再让它执行 |
| **停止生成** | `Esc` | 中断当前生成 |
| **撤销操作** | `Esc Esc` 或 `/rewind` | 撤销上一步 |
| **清空上下文** | `/clear` | 清空对话历史 |
| **语音输入** | `/voice` | 语音模式，信息密度远超打字 |
| **压缩上下文** | `/compact` | 压缩冗余 token |
| **切换模式** | `Shift+Tab` | 在 Code Mode 和 Plan Mode 之间切换 |

## CLAUDE.md 维护策略

CLAUDE.md 不是写完就不管的静态文件，而是一个**持续学习的知识库**。

### 迭代更新原则

```
发现 Claude 犯了一个错误
  ↓
纠正这个错误
  ↓
把「为什么这是错的」写进 CLAUDE.md
  ↓
下次 Claude 不会再犯
```

**实战建议**：
- 每次纠正错误后问自己：「这个错误是 CLAUDE.md 里能预防的吗？」
- 保持精简：控制在 50-200 行，过长反而会被忽略
- 区分 CLAUDE.md（约定性 ~80% 遵守率）和 Hooks（100% 确定性执行）
- 用 `CLAUDE.local.md` 存放个人偏好（如编辑器、格式化习惯），不提交到 Git

## MCP 和工具集成

### 优先安装的 MCP 服务器

| MCP | 用途 |
|-----|------|
| **Playwright** | UI 验证、截图比对、端到端测试 |
| **PostgreSQL / MySQL** | 直接查询数据库 Schema 和数据 |
| **Slack** | Bug 报告、状态通知 |
| **Figma** | 设计稿转代码 |

### CLI vs MCP

> CLI 工具比 MCP 更节省上下文。

当你只需要简单的命令输出时（如 `git log`、`curl`），直接用 `!` 前缀执行 bash 命令比加载 MCP 更高效。MCP 的价值在于**需要持续交互的场景**（如操作浏览器、查询数据库）。

## 提效 Checklist

开始新项目时，按此清单配置可获得最佳体验：

- [ ] 编写 CLAUDE.md，覆盖构建命令、代码规范和架构决策
- [ ] 配置 `/permissions` 白名单，减少确认中断
- [ ] 安装必要的 MCP 服务器（至少 Playwright）
- [ ] 准备好测试命令，写入 CLAUDE.md
- [ ] 创建常用的自定义斜杠命令
- [ ] 配置 Hooks 自动格式化（见高级功能章节）
- [ ] 设置 Git Worktree 用于并行开发
