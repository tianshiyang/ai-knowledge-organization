# 常见坑

下面这些坑专门挂回前面题库里的问题，方便你在面试里成体系地回答。

## 1. 工具太多导致选错

对应题目：

- Agent 如何使用工具？工具选择机制是怎样的？

回答思路：

“我会优先治理工具边界，合并重复工具、重写描述、减少参数，并让返回结果结构化，而不是只靠补 prompt。”

## 2. 状态膨胀

对应题目：

- 状态设计为什么重要？
- Memory 通常如何实现？

回答思路：

“我会把 `messages` 和结构化 `state` 分开，把 `goal`、`plan`、`route`、`tool_results` 等字段显式化，避免所有信息都塞进聊天历史。”

## 3. 死循环

对应题目：

- 如何防止 Agent 死循环？
- 什么是 ReAct？

回答思路：

“我会同时设计最大轮数、最大工具调用数和无进展检测。如果连续多轮没有新增有效信息，就提前停止或降级。”

## 4. Planner 计划不稳

对应题目：

- 常见 Agent 架构有哪些？
- Plan-and-Execute 适合什么场景？

回答思路：

“我会限制计划长度，让 planner 输出结构化步骤，并保留重规划能力，而不是完全相信第一版计划。”

## 5. Router 误分流

对应题目：

- 多 Agent 和单 Agent 的区别是什么？
- Router 和 Supervisor 的边界是什么？

回答思路：

“Router 结果最好结构化，并保留兜底链路。跨域请求明显增多时，就不该继续硬分流，而应该升级协调模式。”

## 6. Critic 空转

对应题目：

- Reflection 有什么价值？

回答思路：

“反思必须建立在明确评审标准上，例如字段完整性、风险覆盖和逻辑一致性。没有检查项的 critic 很容易空转。”

## 7. 子 Agent 上下文失真

对应题目：

- 什么是多 Agent 系统？

回答思路：

“我会把子 Agent 包成接口清晰的工具，明确输入输出和上下文边界，避免一层层自然语言转述。”

## 8. 只改 Prompt，不改系统

对应题目：

- LangChain Agent 是如何工作的？
- 为什么复杂 Agent 要用 LangGraph？

回答思路：

“复杂 Agent 的问题很多不在 prompt，而在工具设计、状态流、控制流和权限边界。只改 prompt 往往治标不治本。”
