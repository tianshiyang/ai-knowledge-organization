# Agent 系统设计题

这页专门承接高频题之后的开放式设计题。重点不是“唯一标准答案”，而是训练一种稳定的回答路径。

## 通用回答模板

建议按下面顺序回答：

1. 需求澄清
2. 边界定义
3. 架构选型
4. 状态设计
5. 工具设计
6. 控制流
7. 失败恢复
8. 评估指标
9. 风险与上线策略

## 题 1：设计一个客服 Agent

### 题目

设计一个企业客服 Agent，支持订单查询、物流查询、退款状态查询和常见 FAQ 问答。

### 推荐回答

这题更适合从 `单 Agent + tools + FAQ/RAG` 起步，而不是一上来就多 Agent。

### 建议架构

- 一个主 Agent
- 读工具：订单、物流、退款、FAQ/RAG
- 写动作：退款申请、人工转接单独确认

### 核心状态

- `messages`
- `user_id`
- `intent`
- `tool_results`

### 关键追问

- 为什么不用多 Agent：因为能力域集中，单 Agent 更稳
- 如何降低幻觉：订单类问题必须查工具
- 如何评估：任务完成率、转人工率、错误答复率、平均时延

## 题 2：设计一个研究 Agent

### 题目

设计一个围绕主题做资料搜集、比较观点并输出总结报告的研究 Agent。

### 推荐回答

这题比客服 Agent 更适合讲成 `ReAct`、`Plan-and-Execute` 或 `Reflection` 的组合。

### 建议架构

- `planner`：拆分研究子问题
- `executor`：搜索、读取、汇总资料
- `critic`：检查来源覆盖和结论一致性
- `writer`：输出最终报告

### 核心状态

- `goal`
- `plan`
- `step_index`
- `tool_results`
- `draft`
- `review`

### 关键追问

- 为什么要规划：研究任务更长、更开放
- 为什么要反思：需要补全证据链和风险项
- 如何控成本：限制搜索轮数、缓存工具结果、限制反思轮次

## 题 3：设计一个企业多 Agent 助手

### 题目

设计一个企业内部助手，支持日程安排、邮件起草、报销咨询、知识检索等复杂任务。

### 推荐回答

可以讲成 `Router + Supervisor + Human-in-the-loop` 的组合，但一定要先说明为什么不能简单把所有工具塞给一个 Agent。

### 建议架构

- `Router`：先做领域分流
- `Supervisor`：只在跨域任务中持续调度
- `Subagents`：日历、邮件、财务、知识库
- `Human-in-the-loop`：写操作、人事审批、外发消息要确认

### 核心状态

- `messages`
- `route`
- `goal`
- `subtasks`
- `subagent_results`
- `approval_required`

### 关键追问

- 为什么需要多 Agent：能力域和权限域都明显分化
- 如何避免过度设计：单域请求只走单链路
- 如何控风险：高风险动作必须经确认节点

## 面试加分点

- 主动说“不是所有问题都应该上多 Agent”
- 能把控制流讲成“节点 + 条件边 + 状态”
- 能讲失败恢复、评估与灰度，而不是只讲 happy path
