# Agent 高频面试题清单

这份题单参考了你给的面试题汇总文章里 `Agent` 相关部分的组织方式，但只保留 Agent 开发最核心、最常问、最适合工程面试的内容。

统一回答模板：

- `问题`
- `面试官想考什么`
- `简答版`
- `展开版`
- `LangChain 1.0 实现点`
- `追问点`

## 1. 什么是 AI Agent？它的核心组件有哪些？

### 面试官想考什么

- 你是否理解 Agent 的本质不是普通聊天
- 你是否知道一个 Agent 最少由哪些部分组成

### 简答版

AI Agent 是一种围绕目标持续决策的系统。它不只是生成一次答案，而是会根据当前状态决定要不要调用工具、要不要继续搜索、要不要修正结果，直到任务结束。

### 展开版

一个工程上最常见的 Agent 通常包含这些核心组件：

- `感知 / 输入`：接收用户问题、环境状态或外部事件
- `推理引擎`：通常由 LLM 负责理解、规划和决策
- `工具 / 行动`：用于访问外部系统，例如搜索、数据库、API
- `状态 / 记忆`：保存当前任务上下文和历史信息
- `结束条件`：决定什么时候停止

如果把它和普通 prompt 问答相比，差别在于：普通 prompt 通常是一问一答，而 Agent 是“目标驱动 + 多步决策 + 中间状态”。

### LangChain 1.0 实现点

最小闭环通常从：

```python
create_agent(model=..., tools=[...], system_prompt=...)
```

开始。复杂控制流再用 `LangGraph` 把状态和节点显式化。

### 追问点

- Agent 和 Workflow 的区别是什么
- Agent 一定要有工具吗
- Agent 为什么不总是比普通工作流更好

## 2. 常见的 Agent 架构有哪些？

### 面试官想考什么

- 你是否知道常见架构的名字和边界
- 你是否会做选型，而不是只会背概念

### 简答版

工程上最常见的 Agent 架构包括：

- `Tool Calling`
- `ReAct`
- `Self-Ask`
- `Plan-and-Execute / Plan-and-Solve`
- `Router`
- `Reflection`
- `Supervisor / Multi-Agent`

另外 `CoT`、`ToT`、`Prompt Chaining` 经常和 Agent 一起被问到，但它们更适合看作推理或提示策略，不完全等同于 Agent 架构。

### 展开版

这些架构可以按复杂度理解：

- 最小起点是 `Tool Calling`
- 需要多轮“查一下再决定下一步”时用 `ReAct`
- 需要把大问题拆小问题时可以谈 `Self-Ask`
- 长任务更适合 `Plan-and-Execute`
- 多领域入口适合 `Router`
- 质量要求高时加 `Reflection`
- 能力域和权限域差异很大时再考虑 `Multi-Agent`

### LangChain 1.0 实现点

- 简单架构优先 `create_agent + tools`
- 复杂编排优先 `LangGraph`

### 追问点

- ReAct 和 Tool Calling 有什么关系
- Self-Ask 和 Plan-and-Execute 有什么区别
- 多 Agent 为什么不是默认起点

## 3. Agent 中的 Memory 通常如何实现？分哪些类型？

### 面试官想考什么

- 你是否理解短期状态和长期记忆的区别
- 你是否知道常见记忆方案

### 简答版

Memory 一般分成短期状态和长期记忆。短期状态负责当前任务执行，长期记忆负责跨会话保存历史信息、偏好或可检索经验。

### 展开版

工程上可以用下面这套口径来讲：

- `短期状态`
  - `messages`
  - `tool_results`
  - `plan`
  - `draft`
- `长期记忆`
  - `conversation buffer`
  - `summary memory`
  - `window memory`
  - `entity memory`
  - `vector-store-backed memory`

很多系统其实不是真的缺“记忆”，而是状态没有设计清楚。面试里最好先把状态和记忆分开说。

### LangChain 1.0 实现点

- 简单场景先用 `messages` 和结构化 `state`
- 长期记忆更适合放到摘要、实体存储或向量检索层
- 不要默认把所有历史都塞回 prompt

### 追问点

- 什么时候不需要长期记忆
- 向量记忆和聊天记录有什么区别
- 如何避免记忆污染

## 4. Agent 如何使用工具？工具选择机制是怎样的？

### 面试官想考什么

- 你是否真的做过工具设计
- 你是否知道模型为什么会选错工具

### 简答版

Agent 会把工具描述、参数定义和当前任务一起交给模型，再由模型决定要不要调用某个工具、调用哪个工具、传什么参数。工具选择稳定性高度依赖工具设计质量。

### 展开版

工具选择机制可以分成三类：

- `LLM 自然语言推理选工具`
- `函数调用 / 结构化工具调用`
- `Router / 显式分流`

设计工具时要注意：

- 名称清晰
- 描述可区分
- 参数少而准
- 返回结构化
- 高风险写操作默认不要直接开放

### LangChain 1.0 实现点

`create_agent` 会把工具暴露给模型。工程上建议工具默认返回 `dict` 或结构化结果，而不是长自然语言。

### 追问点

- 为什么工具太多会变差
- 如何降低错误工具调用
- 写操作为什么要拆成“建议”和“执行”

## 5. 什么是多 Agent 系统？相比单 Agent 有什么优势和挑战？

### 面试官想考什么

- 你是否知道多 Agent 什么时候才真正有必要
- 你是否理解上下文隔离和权限隔离

### 简答版

多 Agent 系统是由多个专长 Agent 协同完成任务的系统。它适合能力域差异大、权限边界强或跨域协作明显的场景，但也会带来更高的编排成本和时延。

### 展开版

相比单 Agent，多 Agent 的优势有：

- 能力边界清晰
- 权限隔离更容易
- 上下文更聚焦
- 团队维护更方便

挑战主要有：

- 编排复杂
- 上下文传递失真
- 成本和时延更高
- 故障面更大

### LangChain 1.0 实现点

常见做法是把子 Agent 包装成 `tool`，由 `supervisor` 统一调度；复杂流程再用 `LangGraph` 显式管理状态流。

### 追问点

- 单 Agent 的上限在哪里
- Router 和 Supervisor 的边界是什么
- 多 Agent 怎么做人审

## 6. 如何评估 AI Agent 的性能？有哪些关键指标？

### 面试官想考什么

- 你是否把 Agent 当工程系统来做
- 你是否知道“过程正确”也要被评估

### 简答版

Agent 评估不能只看最后答案，还要看过程。常见指标包括任务完成率、工具调用正确率、路线正确率、成本、时延和人工接管率。

### 展开版

可以分成两层：

- `离线评测`
  - 任务完成率
  - 工具调用正确率
  - 路由准确率
  - 输出格式正确率
  - 平均步骤数
  - token 成本与时延
- `线上观察`
  - 用户满意度
  - 人工接管率
  - 错误链路分布
  - 风险动作触发率

不同架构还有自己的过程指标。例如：

- Router 看分流准确率
- Reflection 看修订是否提升质量
- Plan-and-Execute 看计划完成率

### LangChain 1.0 实现点

无论是 `create_agent` 还是 `LangGraph`，都要把关键节点、工具结果和结束原因打日志，便于回放和回归测试。

### 追问点

- 如何做离线评测集
- 如何判断模型错了还是工具数据错了
- 如何做版本回归

## 7. Agent 开发中的安全问题有哪些？如何防范？

### 面试官想考什么

- 你是否考虑过真实上线风险
- 你是否知道 Agent 的风险不只是“答错”

### 简答版

Agent 的安全问题包括提示注入、越权调用、错误外部操作、数据泄露和权限滥用。核心防护思路是最小权限、结构化边界、关键动作确认、日志审计和持续评测。

### 展开版

最常见的风险包括：

- `Prompt Injection`
- `Tool Injection / 恶意工具返回`
- `越权访问`
- `敏感信息泄露`
- `错误写操作`
- `高风险动作误执行`

常见防范策略：

- 细粒度权限控制
- 读写工具分层
- 高风险动作加 human-in-the-loop
- 工具输入输出校验
- 实时监控与日志审计
- 红队测试与回放测试

### LangChain 1.0 实现点

真正的安全控制不应只靠 prompt。更稳的方式是把权限、人审和关键分支做成系统层或图节点层能力。

### 追问点

- 为什么发邮件、打款、审批必须确认
- Prompt Injection 为什么不能只靠提示词防
- 如何设计审计日志

## 8. 什么是 ReAct？它的工作流程是什么？

### 面试官想考什么

- 你是否理解 ReAct 的闭环
- 你是否知道它为什么经典

### 简答版

ReAct 是一种“推理 + 行动 + 观察”的 Agent 执行模式。模型先思考下一步要做什么，必要时调用工具，再根据观察结果继续推理，直到任务完成。

### 展开版

它的经典流程是：

`输入 -> Thought -> Action -> Observation -> 再次 Thought -> ... -> Final Answer`

ReAct 的价值在于：

- 让模型在外部信息支持下逐步求解
- 适合多步搜索、查询和动态路径问题

它的风险在于：

- 成本更高
- 容易循环
- 工具错误会放大偏差

### LangChain 1.0 实现点

LangChain 1.0 下不需要把旧版 `create_react_agent` 作为默认入口。新项目更推荐从：

```python
create_agent(model=..., tools=[...], system_prompt=...)
```

开始；当你需要精细控制循环和状态时，再用 `LangGraph`。

### 追问点

- ReAct 和 Tool Calling 的区别是什么
- ReAct 和 Plan-and-Execute 的区别是什么
- 如何防止 ReAct 死循环

## 9. LangChain Agent 是如何工作的？ReAct 是唯一的 Agent 类型吗？

### 面试官想考什么

- 你是否了解 LangChain 1.0 的推荐方式
- 你是否知道 ReAct 不是唯一答案

### 简答版

LangChain Agent 的核心是让模型动态决定何时调用工具，而不是像固定链路那样预先写死步骤。ReAct 很经典，但绝不是唯一的 Agent 形式。

### 展开版

从运行机制来看，LangChain Agent 通常会做这些事：

1. 接收用户输入
2. 把输入、工具描述和上下文交给模型
3. 模型决定直接回答还是调用工具
4. 工具执行后把结果反馈回来
5. 重复这个循环直到结束

但工程上不只有 ReAct 这一条路。常见还有：

- `Tool Calling`
- `Self-Ask`
- `Router`
- `Reflection`
- `Supervisor`

### LangChain 1.0 实现点

统一口径：

- 简单 Agent：`create_agent`
- 复杂工作流：`LangGraph`

### 追问点

- 为什么新项目不推荐旧入口
- LangChain 和 LangGraph 是什么关系
- 什么时候该转到显式图编排

## 10. 什么是 Prompt Chaining？它如何帮助解决复杂任务？

### 面试官想考什么

- 你是否能区分链式工作流和自由 Agent
- 你是否知道复杂任务不一定非要用 Agent

### 简答版

Prompt Chaining 是把复杂任务拆成多个连续提示步骤，让上一步输出成为下一步输入。它适合流程稳定、步骤清晰的复杂任务。

### 展开版

它的优势是：

- 可控性更强
- 更容易测试
- 更容易插入校验

它和 Agent 的区别在于：

- Prompt Chaining 更偏固定流程
- Agent 更偏动态决策

如果你已经知道完整步骤，通常应该优先考虑链式工作流，而不是强行 Agent 化。

### LangChain 1.0 实现点

在 LangChain 1.0 口径下，这类能力可以通过链式组合或 LangGraph 固定节点来实现，不一定要作为 Agent。

### 追问点

- Prompt Chaining 和 Plan-and-Execute 的区别
- 什么场景该用 Workflow，不该用 Agent

## 11. Agent 和 Workflow 有什么区别？

### 面试官想考什么

- 你是否有架构克制能力

### 简答版

Workflow 是预定义步骤，Agent 是动态决策步骤。已知流程优先 Workflow，不确定路径才更适合 Agent。

### 展开版

如果流程稳定，比如“取数 -> 清洗 -> 输出报告”，Workflow 更稳也更便宜。Agent 适合目标明确但执行路径不固定的任务，比如“帮我调研一个主题并总结观点”。

### LangChain 1.0 实现点

- 固定流程：直接链式编排或 LangGraph 固定图
- 动态决策：`create_agent` 或 `LangGraph` 条件边

### 追问点

- 为什么复杂 Agent 往往最后还是图编排
- Workflow 里也能嵌模型节点吗

## 12. 为什么不是所有任务都适合 Agent？

### 面试官想考什么

- 你是否知道 Agent 的成本和边界

### 简答版

因为 Agent 带来灵活性，也带来更高的成本、时延、不确定性和调试难度。流程固定的问题，普通工作流通常更优。

### 展开版

不适合 Agent 的情况包括：

- 输入和输出都很固定
- 流程已经明确
- 时延极其敏感
- 风险高但没有人审和回退

### LangChain 1.0 实现点

工程上建议从最小方案开始，先验证是否真的需要 Agent，再决定要不要引入图编排和多 Agent。

### 追问点

- 什么时候单 Agent 就够了
- 什么时候该从 Agent 回退到 Workflow

## 13. 如何防止 Agent 死循环？

### 面试官想考什么

- 你是否考虑过结束条件和收益判断

### 简答版

要同时设计硬限制和软判断。硬限制包括最大轮数、最大工具调用数、超时；软判断包括重复检测和状态是否有推进。

### 展开版

如果连续几轮都没有新信息、没有新增有效工具结果、没有更接近目标，就应该提前停止、降级或切换到人工。

### LangChain 1.0 实现点

在 `create_agent` 层做最大迭代限制；在 `LangGraph` 层做条件边和退出判断。

### 追问点

- 如何定义“没有推进”
- 结束时如何保留可解释日志

## 14. 什么时候应该加 Human-in-the-loop？

### 面试官想考什么

- 你是否有真实上线意识

### 简答版

高风险、不可逆、外部写操作、成本高的动作，都应该考虑人审。

### 展开版

例如：

- 发邮件
- 发起审批
- 改写数据库
- 财务相关操作

读取类动作可以更自动化，写入类动作要更保守。

### LangChain 1.0 实现点

在图编排中把确认节点显式做出来，而不是只依赖 prompt 里说“请谨慎”。

### 追问点

- 审批节点放在哪一层
- 如何保留审核上下文

## 15. 状态设计为什么重要？

### 面试官想考什么

- 你是否理解复杂 Agent 的稳定性来自状态流

### 简答版

因为复杂 Agent 不是靠多写几个 prompt 稳定下来，而是靠“当前知道什么、做到哪一步、下一步依据什么决策”这些结构化状态稳定下来。

### 展开版

复杂场景常见状态字段包括：

- `messages`
- `goal`
- `plan`
- `step_index`
- `tool_results`
- `route`
- `draft`
- `review`

如果把这些都塞进自然语言对话里，后续调试、重试和评估会越来越难。

### LangChain 1.0 实现点

`LangGraph` 非常适合把状态结构化表达出来。

### 追问点

- `messages` 和结构化 `state` 应该怎么分
- 如何避免状态膨胀
